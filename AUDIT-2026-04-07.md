# z2m_irrigation pre-3.1 audit — 2026-04-07

## Why this audit happened

The user reported a recurring failure mode: the smart irrigation calculator
would call `z2m_irrigation.start_liters` for a zone (e.g., 800 L), and on
checking later they would find the valve still running with 2000–3000 L+
recorded. Sometimes the opposite would happen: a valve would appear to be
"running" but with `flow: 0.00 L/min` and `session_liters` stuck for hours.

This document captures the analysis of `manager.py`, `database.py`, and the
HA supervisor journal logs that led to the v3.1 safety release.

---

## Architecture as audited

- The integration owns its own SQLite database (`z2m_irrigation.db` in HA's
  config dir) for valve totals and a session history table.
- Real-time control flow lives entirely in `manager.py:_on_state()`, the MQTT
  message handler. Flow is integrated over `dt = now - last_ts` whenever an
  MQTT message arrives, accumulating into `session_liters`.
- The "primary failsafe" was a single check inside `_on_state`:
  `if v.session_liters >= v.target_liters: publish OFF; clear target_liters`.
- `scheduler.py` is dead code (`scheduler = None` is hardcoded in
  `__init__.py` since v3.0.0); all scheduling is done externally via HA
  automations and scripts that call `z2m_irrigation.start_liters`.
- Two separate switch entities exist per valve:
  `switch.{topic}` (raw Z2M, source of truth) and
  `switch.{topic}_valve` (custom integration). Both control the same
  physical device.

## Bugs found (ranked by severity)

### #1 — CRITICAL: Volume target lost on HA restart
**File**: `manager.py:_ensure_valve()` (~ line 173)

`Valve` objects were recreated fresh on every HA startup, with only
**lifetime totals** loaded from the DB. The fields `target_liters`,
`session_active`, `session_end_ts`, `session_start_ts`, `current_session_id`
were NEVER persisted or restored.

**Consequence**: After any HA restart (Core update, OS update, crash) while
a valve was physically open, the next MQTT `state: ON` message would create
a brand-new in-memory session with `target_liters=None`. The failsafe block
`if v.target_liters and v.target_liters > 0:` would then never trigger, and
the valve would run unbounded.

**Likelihood it caused user-reported runaway**: HIGH. Any unscheduled
restart mid-irrigation results in this state.

### #2 — CRITICAL: No time-bound failsafe for volume runs
**File**: `manager.py:_on_state()` (~ line 238)

The volume failsafe ran ONLY inside `_on_state()`, which is the MQTT message
handler. There was no periodic timer independent of MQTT. Compare with
`start_timed()` (~ line 475) which DID set `async_call_later(...)` as a
backup — that was missing entirely from `start_liters()`.

**Consequence**: If the device stopped sending MQTT messages (Zigbee
dropout, Z2M restart, broker hiccup, device sleep, dead battery), the
failsafe would never fire regardless of how much real-world time had passed.

**Real-world observation**: Lilly Pilly valve was observed live in the
journal during the audit, with debug logs showing `2.38/567.00 L (0.4%),
flow: 0.00 L/min` repeating identically across 3.5+ hours. The session was
"active" in HA's view but no liter accumulation was happening. Eventually
identified that the device had been silent for 13 hours (battery was 80%
yesterday but stale by then).

### #3 — CRITICAL: Runaway integration after primary failsafe fires
**File**: `manager.py:_on_state()` (~ line 244)

```python
if v.session_liters >= v.target_liters:
    self._schedule_task(self.async_turn_off(topic))
    v.target_liters = None  # Clear target to prevent repeated triggers
    return
```

When the failsafe fired:
- It published OFF **once** with no retry, no PUBACK verification, no
  escalation.
- It immediately cleared `v.target_liters = None`.
- `session_active` stayed `True` (and `v.state == "ON"`) until/unless the
  device sent an `state: OFF` message.

**Consequence**: If the device ignored the OFF (Zigbee delivery failure,
firmware glitch, valve mechanically stuck), `session_active` stayed `True`,
flow continued to integrate, `session_liters` kept climbing — but **no
further failsafe ever fired** because `target_liters` was now `None`.

**This is the most likely root cause of the user's "800 L target →
2000–3000 L actual" complaint.** Once the failsafe fires and the device
doesn't actually close, there's no second line of defense.

### #4 — HIGH: No retry / verification of OFF command
**File**: `manager.py:async_turn_off()` (~ line 430)

```python
async def async_turn_off(self, topic: str) -> None:
    await mqtt.async_publish(self.hass, f"{self.base}/{topic}/set",
                             json.dumps({"state": "OFF"}), qos=1)
```

A single `async_publish` with QoS 1, which only guarantees broker delivery,
not Z2M → Zigbee → device delivery. No retry loop, no verification, no
escalation.

### #5 — HIGH: Failsafe gated on stale `v.state`
**File**: `manager.py:_on_state()` (~ line 240)

The failsafe block was guarded by `if v.state == "ON":`. `v.state` is
HA's cached view of the valve, only updated when MQTT messages with a
`state` key arrive. If the device's last MQTT message had no `state`
field (e.g., a flow-only update), `v.state` kept its previous value.
Mostly correct, but it means the failsafe was always one MQTT message
behind reality. Compounds with #2 and #3.

### #6 — MEDIUM: Session-end logging duplicate / state-flap
**Observed during audit**: when the user manually toggled Front Garden
off, the journal showed:

```
🛑 [MANAGER] Session ending for Front Garden: 0.00min, 0.00L, 1.67lpm
Session ended: Front Garden_1775553443.00175 - 0.00min, 0.00L
Saved totals for Front Garden: +202.77L, +121.70min
```

The `0.00min, 0.00L` ending log fired on what was likely a SECOND, brief,
freshly-started session (probably the device flapping `ON → OFF → ON → OFF`
in response to receiving the OFF command). The previous correct session
was saved with the right totals (`+202.77L`), but a phantom 60-millisecond
session also got recorded.

**Consequence**: Bad session statistics in the DB (lots of zero-duration
sessions). Not a safety issue. **Deferred to v3.1.1**.

### #7 — MEDIUM: Race in clearing target_liters before publish
**File**: `manager.py:_on_state()` (~ line 248)

```python
self._schedule_task(self.async_turn_off(topic))
v.target_liters = None
```

`_schedule_task` enqueues `async_turn_off` for the event loop; it doesn't
run synchronously. `target_liters = None` happens immediately. If a second
`_on_state` callback fires before the OFF is published (very possible —
MQTT updates can arrive in bursts), the second callback sees
`target_liters = None` and skips the failsafe entirely.

**Fixed in v3.1**: the new shutoff state machine sets
`shutoff_in_progress = True` synchronously, which is checked at the top of
the failsafe block (`if v.state == "ON" and not v.shutoff_in_progress:`).

### #8 — LOW: Two switch entities for the same physical valve
**File**: `switch.py`

`switch.{topic}` (raw Z2M) and `switch.{topic}_valve` (custom integration)
both exist for every valve, controlling the same physical device. UX
confusion only; not a safety issue. **Deferred — would change user-visible
entity IDs, requires a migration plan.**

### #9 — LOW: Periodic refresh doesn't check active sessions
**File**: `manager.py:_periodic_refresh_time_metrics()`

Runs every 15 minutes but only updates 24h/7d sensors. It would have been
the natural place for a guardrail check, but instead the v3.1 release
adds a separate dedicated `_guardrail_tick()` running every 30s.

## Live observations during the audit

### Lilly Pilly stuck in volume run
At the time of the audit (around 19:15 local), the journal showed:

```
2026-04-07 15:36:33 ... Volume run progress for Lilly Pilly: 2.38/567.00 L (0.4%), flow: 0.00 L/min
2026-04-07 15:37:25 ... Volume run progress for Lilly Pilly: 2.38/567.00 L (0.4%), flow: 0.00 L/min
... (continuing identically for 3.5+ hours) ...
2026-04-07 19:15:12 ... Volume run progress for Lilly Pilly: 2.38/567.00 L (0.4%), flow: 0.00 L/min
```

State queries via the HA REST API revealed that
`sensor.lilly_pilly_flow.last_updated` was 13 hours stale — the device had
been completely silent since the morning. The session was "active" in HA's
view but the device was effectively offline. This is the **stuck-flow** /
**MQTT silence** failure mode that v3.1 Layers 2 and 3 are designed to
catch.

### Tesla automation noise (separate issue)
Also observed: `automation.tesla_adjust_charging_rate_to_solar_production`
was firing multiple times per second, generating ~480k automation triggers
and ~660k call_service events in the recorder. This is unrelated to
z2m_irrigation but contributed to general HA performance issues. Tracked
separately.

## Why "no hard time cap"

When discussing fixes, the question of a single hard ceiling (e.g.,
"no run can ever exceed 2 hours") was considered and rejected by the user.
The problem with a hard cap is that legitimate large runs exist (e.g., a
hedge zone with `1.4 × 60 L/mm × 6 mm = 504 L` baseline, which can climb
above 800 L on hot dry weeks).

The v3.1 design instead uses **layered soft guardrails**, each catching a
different failure mode, with no fixed time ceiling. Any one guardrail
firing prevents a runaway. Most failure modes are caught by 2+ layers.

---

## Fix philosophy

- **Surgical, not a rewrite.** All v3.1 changes are contained to four files
  with no schema changes. The core control loop is preserved.
- **Defense in depth.** Five overlapping layers, no single point of failure.
- **Loud failures.** Every guardrail trigger fires both an HA event (for
  automation hooks) and a persistent notification (for user visibility).
- **Idempotent and self-cleaning.** The shutoff state machine cannot be
  double-triggered, cleans up after success, and gives up gracefully after
  the retry budget is exhausted.
- **Safest possible startup behavior.** On HA restart, any orphaned
  in-flight session results in a force-OFF + DB cleanup + user notification.
  We do NOT try to "resume" the session because we cannot know the physical
  state of the valve.
